# Vastu Griha Asset Library Documentation

This document defines the folder structures, naming rules, file specifications, optimization requirements, and licensing guidelines for the production asset management system powering Vastu Griha inside BanjaraBazaarOS.

---

## 1. Folder Structure

All core assets are organized inside `/apps/vastu-griha/assets/` under strict category directories. Any new asset type must belong to one of these ten standard directories:

```text
/apps/vastu-griha/assets/
├── asset-manifest.json     # Master index registry
├── rooms/                  # Full-room layout definitions (e.g., bedroom, kitchen)
├── furniture/              # Internal furniture layout symbols (e.g., bed, sofa)
├── decor/                  # Decorative items (e.g., mirror, clock, painting)
├── plants/                 # Indoor/outdoor plant symbols (e.g., tulsi, money plant)
├── utilities/              # Electrical, water, or mechanical utilities (e.g., water tank)
├── structure/              # Structural boundaries (e.g., main door, walls, windows)
├── vastu/                  # Corrective Vastu remedies and tools (e.g., copper pyramids, helixes)
├── symbols/                # Sacred/Vedic energetic geometric icons (e.g., swastika, om)
├── backgrounds/            # Grid systems, patterns, or plot background grids
└── avatars/                # Customer and expert consultant profile icons
```

---

## 2. File Naming Conventions

To ensure the asset loader can resolve URLs reliably and prevent case-sensitive mismatches in production environments:
* **Lower Case Only**: Use lower-case letters and numbers. Do not use uppercase letters.
* **Underline Separators**: Use underscores (`_`) to separate words. Do not use spaces, hyphens, or special characters.
  * *Correct*: `main_door.svg`, `water_tank.svg`
  * *Incorrect*: `main-door.svg`, `water tank.svg`, `MainDoor.SVG`
* **Semantic Names**: Keep names concise, starting with the primary object followed by details if necessary (e.g., `expert_avatar.svg`, `sofa_3seater.svg`).

---

## 3. Supported File Formats

1. **SVG (Scalar Vector Graphics)**: *Highly Recommended (90%+ of catalog)*
   * Used for all blueprints, room templates, icons, symbols, and remedies.
   * Enables responsive scaling, canvas coordinate placement, and styling hooks via CSS (e.g., modifying `stroke="currentColor"` dynamically in the canvas editor).
2. **PNG (Portable Network Graphics)**:
   * Permitted only for photo-realistic backdrops, complex textured avatars, or raster rendering where vector coordinates are not required.
   * Must include a transparent channel (`alpha`) and be saved at high pixel densities.

---

## 4. Recommended Sizes & Specifications

### Vector Assets (SVG)
* **ViewBox**: Always use a standard square viewBox of `0 0 100 100` or double-size `0 0 200 200`. This enables uniform canvas math.
* **Responsive Styling**:
  * Set `fill="none"` or `fill="currentColor"` appropriately.
  * Set line strokes to `stroke="currentColor"` rather than hardcoded colors so the application can change line colors dynamically between light/dark themes.
  * Set a standard baseline stroke thickness (e.g., `stroke-width="2"` or `stroke-width="1.5"`).

### Raster Assets (PNG)
* **Standard Resolution**: 512px x 512px (for avatars, small elements) or 2048px x 2048px (for plot backdrop layouts).
* **Color Profile**: sRGB.

---

## 5. Optimization & Compression Standards

All assets must be run through an optimization pass before committing to the repository to keep the application fast and lightweight:

* **SVG Optimization (SVGO)**:
  * Remove redundant editor metadata (e.g., Adobe Illustrator or Inkscape comments/namespaces).
  * Minify XML paths.
  * Combine path elements where possible to decrease file size.
* **PNG Compression**:
  * Compress using lossless algorithms like `PNGOUT` or `TinyPNG`.
  * Strip unnecessary metadata chunks (e.g., EXIF camera data).

---

## 6. Licensing & Source Policy

To maintain enterprise integrity and support production scaling, the library enforces a strict legal compliance check:

* **No Copyrighted Elements**: Do not use proprietary icons, stock illustrations, or logos without an explicitly documented license.
* **No AI-Generated Work**: Do not commit assets generated by AI engines (Midjourney, DALL-E, etc.) to ensure clear intellectual property provenance.
* **Attribution**: Every asset must be fully registered in `asset-manifest.json` with its:
  * `license` field: (e.g., `MIT`, `Creative Commons CC-BY-4.0`, or `Proprietary` if owned internally).
  * `source` field: Link to original source or labeled `BanjaraBazaarOS Internal` for proprietary works created by the development team.
